# 技术决策文档

## 服务端构建策略选择

### 问题背景

在 Electron 应用中集成 NestJS 后端服务时，我们面临两种主要的构建策略选择：

1. **使用 `nest build`**: NestJS 官方提供的构建命令
2. **使用 Webpack 打包**: 将所有代码打包成单个文件

### 技术对比

| 方面 | nest build | webpack 打包 |
|------|------------|-------------|
| **输出结构** | 保持模块结构，多个 .js 文件 | 单个打包文件 |
| **构建速度** | 快速 | 相对较慢 |
| **文件大小** | 较大（包含源码映射） | 较小（单文件） |
| **调试体验** | 优秀（清晰的堆栈跟踪） | 一般（需要 source map） |
| **依赖兼容性** | 高（原生模块友好） | 可能有问题 |
| **错误排查** | 容易定位问题 | 相对困难 |

### 最终决策：选择 `nest build`

#### 决策原因

1. **原生模块兼容性**
   - 项目使用了 `better-sqlite3` 等原生 Node.js 模块
   - Webpack 打包可能破坏这些模块的二进制绑定
   - `nest build` 保持原始模块结构，确保兼容性

2. **开发体验优先**
   - 清晰的错误堆栈跟踪
   - 便于调试和问题定位
   - 减少构建配置的复杂性

3. **Electron 环境特点**
   - Electron 应用通常不受文件大小限制
   - 本地运行，网络传输不是考虑因素
   - 更注重稳定性和可维护性

4. **构建效率**
   - `nest build` 构建速度更快
   - 减少了 webpack 配置维护成本
   - 降低了构建失败的风险

#### 实施策略

1. **依赖优化**
   ```javascript
   // 过滤生产依赖，排除开发工具
   const excludePatterns = [
     '@nestjs/cli',
     '@nestjs/schematics', 
     '@nestjs/testing',
     '@types/',
     'eslint',
     'prettier',
     'jest',
     'supertest',
     'ts-',
     'typescript',
     'webpack'
   ];
   ```

2. **模块结构保持**
   - 直接复制 `dist` 目录到 `nodeServer`
   - 保持 NestJS 的模块组织结构
   - 确保所有依赖关系正确

3. **资源管理**
   - 自动复制数据库文件
   - 处理配置文件
   - 创建简化的 `package.json`

### 备选方案

如果未来需要考虑文件大小或有其他特殊需求，可以考虑以下方案：

1. **混合策略**
   - 核心业务逻辑使用 webpack 打包
   - 原生模块保持独立

2. **分层打包**
   - 按模块分别打包
   - 减少单个文件复杂度

3. **动态加载**
   - 按需加载模块
   - 减少启动时间

### 监控指标

为了验证决策的正确性，我们需要监控以下指标：

- **构建时间**: 完整构建所需时间
- **应用启动时间**: 从启动到服务可用的时间
- **内存使用**: 运行时内存占用
- **错误率**: 构建和运行时错误频率
- **开发效率**: 问题定位和修复时间

### 结论

基于当前项目的特点和需求，选择 `nest build` 是最合适的构建策略。这个决策优先考虑了开发体验、稳定性和维护性，符合 Electron 桌面应用的特点。

随着项目的发展，我们会持续评估这个决策的有效性，并在必要时进行调整。
